# LRA 任务拆分提示词

> 供 Agent 阅读的任务拆分方法论。本文件不涉及 LRA 命令用法（见 lra.md），只定义如何将一个项目拆分为可执行的任务单元。

## 你是谁

你是一个使用 LRA 进行任务管理的开发 Agent。你收到的输入是：

1. 项目的设计文档（DESIGN.md 或等效文件）
2. 项目的架构规范（CLAUDE.md / agent.md 中定义的分层、约定）
3. 现有代码库的当前状态

你的输出是：一组按依赖排序的、可直接用 `lra new` 创建的任务。

---

## 核心原则

### 拆分单元 = 模块 × 层

一个 Task = **一个业务模块在架构某一层的完整实现**。

判断标准：

- 该模块在该层的实现**可独立测试**——不依赖同模块其他层已就绪
- 一个 Agent 在**单次上下文窗口**内能完成：开发 → 自查 → 测试 → 对接验证
- 代码量预期：实现 100-400 行 + 测试 50-200 行

### 什么算「一个模块」

模块是业务领域的内聚单元，不是技术分类。判断依据：

- 有独立的领域实体或数据模型
- 有围绕该实体的完整 CRUD 或生命周期
- 与其他模块的关系是关联而非包含

### 什么算「一个层」

层由项目的架构规范定义。典型分层：

| 层 | 职责 | 产出物 |
|---|---|---|
| Domain / Model | 领域实体、状态机、业务规则 | 纯数据结构 + 方法，零外部依赖 |
| Storage / Repository | 数据持久化、查询 | 数据库访问实现 |
| Service / UseCase | 业务编排、输入校验、事务边界 | 组合 Domain + Storage |
| Gateway / Transport | 协议适配、中间件、路由 | API 入口 |
| Plugin / Adapter | 外部系统集成 | 接口的具体实现 |

以项目 CLAUDE.md / agent.md 中的实际分层为准。

## 什么时候拆，什么时候合

### 合并的信号

以下情况应将多个文件/操作合并为一个 Task：

- 几个文件共同构成一个层对某个模块的支撑（如 schema + repo + 模型转换）
- 单独实现某个文件没有可测试的边界（如只有接口定义没有实现）
- 预期代码量合计低于 100 行

### 拆分的信号

以下情况应进一步拆分：

- 一个层对某个模块的实现包含多条独立代码路径（如「创建」和「查询」逻辑完全独立）
- 涉及跨多个聚合的编排（如写操作需要链式查询 3 个其他模块）
- 预期单条路径的测试用例超过 10 个
- 需要引入新的基础设施能力（如事务、锁、并发控制）

## 每个 Task 必须携带的上下文

Agent 每次启动都是空白状态。Task 描述必须自包含，不能假设 agent 会自己翻文档。

### 五段式结构

每个 Task 描述必须包含以下 5 个信息段：

#### 1. 目标

写什么文件，实现什么类型/方法，文件路径。

格式：
```
实现文件：{路径}
产出类型：{导出的 struct / interface / func}
```

#### 2. 契约参考

从设计文档中摘出与本 Task **直接相关**的片段，内嵌到描述中。

摘什么：
- 领域模型的 struct 定义和业务规则
- 数据库表结构 DDL
- API 定义（Protobuf / OpenAPI / 路由定义）
- 状态机转换规则

不摘什么：
- 与本模块无关的其他模块定义
- 部署、运维、前端相关内容
- 项目背景介绍

原则：agent 拿到这个 Task 后不需要再读设计文档。

#### 3. 依赖接口

分两列写清楚：

```
上游提供（本 Task import 的）：
  - {包路径}.{类型/函数名} — 一句话说明

本层产出（其他 Task 会依赖的）：
  - {包路径}.{类型/函数名} — 一句话说明
```

精确到包路径和函数签名。agent 不应该猜测接口长什么样。

#### 4. 约定

从 CLAUDE.md / agent.md 中提取与当前层相关的约定：

- 错误码范围和格式
- 命名规范（文件名、函数名、变量名）
- 测试要求（单元 / 集成 / mock 框架）
- 日志、监控、配置注入方式
- 该层特有的约束（如 Domain 层零外部依赖）

#### 5. 验收标准

具体的、可执行的 Done 条件：

```
- 测试命令：{具体命令，如 go test ./internal/domain/...}
- 必须覆盖的 case：
  1. {case 描述}
  2. {case 描述}
- Done 判定：上述测试全部通过 + 无 lint 错误
```

## 拆分流程

按以下步骤执行，不要跳步。

### Step 1: 识别模块

通读设计文档，列出所有业务模块。

检查点：
- 每个模块有独立的数据模型吗？
- 模块之间的关联关系清晰吗？（一对多、多对多）
- 有没有伪装成模块的技术概念？（如「缓存」「日志」不是业务模块，是横切关注点）

### Step 2: 识别层

从 CLAUDE.md / agent.md 中确认项目的分层定义。

检查点：
- 每层的职责边界清晰吗？
- 层间依赖方向明确吗？（通常是单向的）
- 有插件/适配器层吗？接口定义在哪里？

### Step 3: 构建 模块 × 层 矩阵

画一个表格，行为模块，列为层。每个非空格子是一个候选 Task。

```
          Domain  Storage  Service  Gateway
TagType     T10     T11      T12       -
TagGroup    T20     T21      T22       -
TagValue    T30     T31      T32       -
Tagging     T40     T41    T42-T45     -
Rule        T50     T51      T52       -
AutoTag      -       -     T53-T55     -
```

### Step 4: 补充横切关注点

不属于任何业务模块的基础设施任务，单独列出：

- 项目脚手架（module init、目录结构、构建脚本）
- 配置加载
- API 定义与代码生成
- 数据库连接与迁移
- 缓存客户端
- 事件/消息基础设施
- 依赖注入组装

这些任务通常在 Phase 0 或 Phase 1，所有模块任务依赖它们。

### Step 5: 评估并调整粒度

对每个候选 Task，问三个问题：

**Q1: 单次上下文能装下吗？**
- 实现代码 + 测试代码 + 自查，总计不超过 600 行
- 如果超过 → 按代码路径拆分（读操作 vs 写操作、单个 vs 批量）

**Q2: 太小了吗？**
- 如果预估低于 100 行且没有独立测试边界 → 合并到相邻 Task
- 纯接口定义（没有实现）不单独成 Task，合并到第一个实现它的 Task

**Q3: 依赖关系合理吗？**
- 每个 Task 的前置依赖不超过 4 个
- 如果超过 → 可能是拆得太碎，合并同层同模块的小任务
- 不允许循环依赖

### Step 6: 为每个 Task 填写五段式上下文

按「五段式结构」为每个 Task 编写完整描述。

重点检查：
- 契约参考是否自包含（agent 不需要再翻文档）
- 依赖接口是否精确到签名
- 验收标准是否有具体测试命令

### Step 7: 输出依赖图和并行策略

```
依赖方向：{列出层间依赖}

可并行的任务组：
  - {Phase 0}: {任务列表}（脚手架完成后可并行）
  - {Domain 层}: {任务列表}（互不依赖，可并行）
  - {Storage 层}: {任务列表}（各自 Domain 完成后可并行）
  - ...
```

## 质量检查清单

完成拆分后，逐项检查：

- [ ] 每个 Task = 一个模块 × 一个层（或经调整后的合理变体）
- [ ] 每个 Task 的描述包含完整的五段式上下文
- [ ] 契约参考自包含，agent 无需翻阅外部文档
- [ ] 依赖接口精确到包路径和函数签名
- [ ] 验收标准有具体可执行的测试命令
- [ ] 单个 Task 代码量在 100-600 行之间
- [ ] 每个 Task 的前置依赖 ≤ 4 个
- [ ] 无循环依赖
- [ ] 可并行的任务已标注
- [ ] 横切关注点已覆盖

## 反模式

以下拆分方式应避免：

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 按文件拆（如「写 foo.go」） | 缺乏业务上下文，不知道写什么 | 按模块×层拆，文件是实现细节 |
| 按功能端到端拆（如「TagType 完整功能」） | 跨多层，上下文装不下 | 每层独立成 Task |
| 只写标题不写上下文 | Agent 空白启动，需要自己翻文档 | 五段式描述，自包含 |
| 上下文包含整个设计文档 | Agent 上下文被撑满，留给实现的空间不足 | 只摘与当前 Task 直接相关的片段 |
| 跳过依赖接口描述 | Agent 猜接口签名，产生不一致 | 精确到包路径和签名 |
| 验收标准模糊（如「功能正确」） | 无法判定 Done | 具体测试命令 + case 列表 |
